# Phase 2: 仕様書（spec.md + 周辺ドキュメント）

入力は承認済みの prfaq.md。PRFAQ の顧客体験を、実装可能な仕様に翻訳する。
進行の規範（インタビュー・保存先）は SKILL.md のものをそのまま適用する。

## 手順

### 1. スコープの確定

インタビューで in / out を切る。

- PRの最重要ベネフィットに直結しない要素は out に置き、理由を一行添える。
- PRFAQ に残った「仮説」「未回答」をここで解消するか、検証タスクとして spec に明記するかをユーザーに問う。

**完了条件**: in / out の一覧があり、out の各項目に理由が付いている。

### 2. 仕様の本体

spec.md に以下を書く。

- **ユーザーストーリーと受入条件**: PRの「顧客体験」の各ステップをストーリーに分解する。受入条件は実行して確認できる形（Given/When/Then など）で書く。
- **ドメインモデル**: 登場する概念・状態・遷移。不正な状態が表現できない型設計の方針まで。
- **非機能要件**: この機能で実際に問題になるものだけ（性能目標、権限、データ保持）。一般論のセキュリティ項目の列挙はしない。
- **エッジケースとエラー**: 各ストーリーの失敗系。ユーザーに何が見えるかまで書く。

**完了条件**: すべての受入条件が実行可能な検証手順を持ち、各ストーリーが PRFAQ のベネフィットまたはFAQの回答に遡れる。

### 3. 周辺ドキュメント

仕様の性質に応じて、必要なものだけ作る。該当しなければ作らない。

| 仕様に含まれるもの | 作るドキュメント |
|---|---|
| 外部公開・チーム間契約となるAPI | OpenAPI 定義（openapi.yaml） |
| 複数案から選んだ技術・アーキテクチャ判断 | ADR（`docs/adr/`、日時とモデル名つき） |
| 画面を持つ機能 | 画面フロー・状態一覧 |
| 永続化するデータ | スキーマ定義・マイグレーション方針 |

**完了条件**: 上の表の各行について「作った」か「該当しない」かを判定済みで、作ったものは spec.md からリンクされている。

### 4. 承認ゲート

spec.md と周辺ドキュメントをユーザーに提示し、レビューを受ける。

**完了条件（＝Phase 2 の承認ゲート）**: 未解決の論点が残っていないか、残っている場合は検証タスクとして明記された状態で、ユーザーの承認を得ている。承認後、[PLAN.md](PLAN.md) を読んで Phase 3 を開始する。
