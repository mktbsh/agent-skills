# Phase 3: 実装計画（plan.md）

入力は承認済みの prfaq.md と spec.md。仕様を、依存関係つきの実装チケット列に落とす。
進行の規範（インタビュー・保存先）は SKILL.md のものをそのまま適用する。

## 手順

### 1. トレーサーバレットの特定

最初に実装するのは、**最もリスクの高い仮説を端から端まで貫く最薄のスライス**である。

- PRFAQ の内部FAQ「失敗するとしたら何が原因か」と、spec に残った検証タスクからリスクを拾う。
- そのリスクを実機で確かめられる最小の縦切り（UI→ロジック→永続化を一本通す）を第1チケットにする。

**完了条件**: 第1チケットが「動くと何の仮説が検証されるか」を明記している。

### 2. チケット分解

spec の各ストーリーをチケットに分解し、plan.md に書く。各チケットは以下を持つ。

- **ゴール**: 完了時に何が動くか（作業内容ではなく観察可能な結果で書く）。
- **完了条件**: spec の受入条件のうち、このチケットで満たすもの。
- **ブロック関係**: 先行して完了が必要なチケット。
- **触る範囲**: 影響するモジュール・ファイル領域の見当。

**完了条件**: spec のすべての受入条件がいずれかのチケットに割り当てられ、未割り当てがない。

### 3. 順序とリスクの検算

- ブロック関係をたどり、循環がないこと、トレーサーバレットが先頭にあることを確認する。
- リスクの高いチケットほど前に置く。後ろに残った高リスクがあれば、前倒しできないかユーザーに問う。

**完了条件**: チケットの実行順が一意に定まり、その順で進めた場合に「動くもの」が各マイルストーンで存在する。

### 4. 承認ゲート

plan.md をユーザーに提示し、プロジェクトのトラッカー規約（ローカル markdown トラッカー、GitHub Issues など）があればチケットを登録する。

**完了条件（＝Phase 3 の承認ゲート）**: ユーザーの承認を得て、チケットがトラッカーに登録された状態。以降の実装はチケット単位で進める。
