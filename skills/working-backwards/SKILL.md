# Working Backwards（PR/FAQ）起点の開発プロセス

解決策からではなく、顧客体験の完成形から逆算する。
発売日に出す架空のプレスリリース（PR）と想定問答（FAQ）を実装より先に書き、そこから仕様書・実装計画へ降りていく。

PRFAQ は承認を取るための文書ではなく、**アイデアを殺すための道具**である。
説得力のあるPRが書けないアイデアは、作っても顧客に響かない。書く段階で潰れるのは成功であり、最も安い失敗である。

## パイプライン

| フェーズ | 成果物 | 手順書 |
|---|---|---|
| 1. PRFAQ | prfaq.md | 本ファイル |
| 2. 仕様書 | spec.md + 周辺ドキュメント | [SPEC.md](SPEC.md) |
| 3. 実装計画 | plan.md | [PLAN.md](PLAN.md) |

各フェーズはユーザーの**承認ゲート**で区切る。成果物への承認を得てから、次フェーズの手順書を読んで開始する。
次フェーズの手順書は承認前に読まない — 目の前のフェーズに全力を注ぐ。

## 進行の規範（全フェーズ共通）

- インタビュー形式で進める。質問は一度に1問、回答を受けてから次を聞く。各質問には自分の推奨案を添える。
- 環境（コードベース、ドキュメント、Web）から調べられる**事実**は聞かずに自分で調べる。**判断**（顧客接点の実感、事業判断、優先順位）はユーザーに聞く。
- ユーザーと対話できる状況で `grilling` スキルが利用可能なら、インタビューの開始時に起動して併用する。
- 成果物の保存先はプロジェクト規約に従う。規約がなければ `docs/<機能名>/` 配下。

## ブランチ判定

- アイデアから始める → 手順1から順に進む。
- 既存のPRFAQをレビューする → 手順1は適用外。手順2〜3の規範と完了条件をチェックリストとして適用し、手順4の判定を出す。レビューでは、PR・FAQを問わず出典のない断定を「仮説」とみなして規範を適用する。

## Phase 1: PRFAQ

成果物は prfaq.md 1ファイル（手順1の質疑記録 + PR 1ページ + FAQ 5ページ以内）。

### 1. 顧客の5つの質問

PRを書く前に、インタビューで以下の5問を埋める。

1. 顧客は誰か？
2. 顧客のどんな課題・機会に取り組むのか？
3. 最も重要な顧客ベネフィットは何か？（1つに絞る）
4. 顧客がそれを望んでいると、どうやって知ったか？
5. 顧客体験はどのようなものになるか？

**完了条件**: 5問すべてに回答があり、各回答に次のいずれかのラベルが付いている。

- **事実**: 出典を示せる観測・計測・調査結果。ユーザーが下した決定（価格方針、対象範囲など）も「事実（出典: ユーザーの決定）」として扱う。
- **仮説**: 回答はあるが根拠を示せない推測。複数の事実から導いた結論・解釈もここに含める。
- **未回答**: ユーザー自身も答えを持っていない。推測で埋めず、手順4で検証課題として扱う。

### 2. プレスリリース（1ページ）

未来の発売日の日付で、次の構成で書く。

- **見出し**: 顧客ベネフィットを一文で。見出しだけ読んで顧客が欲しがるかが決まる。
- **サブ見出し**: 誰のための製品で、何が得られるか。
- **要約段落**: 製品と発売の概要。ここまでで読むのをやめる読者に全体が伝わること。
- **課題段落**: 顧客の視点で課題を語る。
- **解決段落**: 製品がどう課題を解くか。
- **提供者の言葉**: 責任者の引用。なぜ作ったか。
- **顧客体験**: 使い始めから価値を得るまでの流れ。
- **顧客の言葉**: 架空の顧客の引用。機能ではなくベネフィットを語らせる。
- **行動喚起**: 顧客が次に取る行動。

規範:

- 一般の読者に通じる言葉だけで書く（社内用語・技術用語が必要になったら、それは顧客の言葉で価値を説明できていない兆候）。
- 形容詞で盛らず、具体的な数字と体験で語る。「劇的に速い」ではなく「3時間の作業が5分になる」。裏付けの数字がない主張は、数字を作らず体験の描写で語る。
- 手順1で「仮説」ラベルが付いた主張は、PRでも仮説として扱い、断定の根拠にしない。

**完了条件**: 1ページに収まり、見出し単体で顧客ベネフィットが伝わる。

### 3. FAQ

外部と内部に分けて書く。**最も答えにくい質問から書く** — FAQの価値は質問の答えにくさに比例する。
回答には手順1と同じラベル（事実/仮説/未回答）を付ける。

- **外部FAQ**（顧客が聞く質問）: 価格、入手方法、既存の代替手段との違い、制約・できないこと。
- **内部FAQ**（意思決定者が聞く質問）: 市場規模、経済性（単価・コスト構造）、技術的実現性と依存関係、なぜ我々がやるのか、なぜ今か、このアイデアが失敗するとしたら何が原因か。

**完了条件**: 「このアイデアを潰しうる質問」が少なくとも1つ含まれ、正面から答えられている。答えられない場合は「未回答」と明記し、それを次の検証課題として挙げている。

### 4. 判定

1. PRを顧客として読み直し、これを欲しいか自答する。
2. 「仮説」ラベルと「未回答」を数え、検証の優先順位を付ける。
3. 三択で判定し、根拠を添える。選定基準:
   - **進める**: 潰しうる質問と最重要ベネフィットが、いずれも事実で裏付けられている。
   - **仮説を検証してから**: 潰しうる質問または最重要ベネフィットに、仮説・未回答が残っている。
   - **やらない**: アイデアを否定する事実（課題の不在、ベネフィットの不成立、経済性の破綻など）が得られた。エビデンスの不足は「検証してから」であり、「やらない」の根拠には否定の事実が要る。

「やらない」は失敗ではなく、このプロセスの成果である。

**完了条件（＝Phase 1 の承認ゲート）**: 三択の判定・根拠・検証すべき仮説と未回答の一覧をユーザーに提示している。出口は判定ごとに異なる。

- **進める**: prfaq.md への承認を得て、[SPEC.md](SPEC.md) を読んで Phase 2 を開始する。
- **仮説を検証してから**: 検証課題の一覧までが成果物。パイプラインはここで止め、検証結果が出たら手順1に戻って再判定する。
- **やらない**: 判定根拠と未検証の論点を記した prfaq.md が最終成果物。パイプラインはここで完了する。
